### Шаг 1. Разработка инференс-модуля

Инференс-модуль реализован в двух вариантах для гибкости:

1. **Inference API (FastAPI)** – синхронный REST-сервис, принимающий CSV с сырыми транзакциями и возвращающий:
   - Прогноз спроса (`/predict`)
   - Оптимальные цены и ожидаемую прибыль (`/optimize`)

   Модуль загружает финальную модель `catboost_model_tuned.cbm` и выполняет полный пайплайн предобработки, разработанный в ЛР3.

2. **Worker (Python)** – асинхронный обработчик задач из очереди RabbitMQ. Реализует ту же бизнес-логику, что и Inference API, но сохраняет результаты в PostgreSQL и кеширует в Redis.

Оба компонента используют общий код предобработки и оптимизации, что обеспечивает консистентность результатов.

---

### Шаг 2. Интеграция компонентов

Система объединяет следующие сервисы:

- **Frontend** (React + TypeScript) – пользовательский интерфейс для загрузки CSV и отображения результатов.
- **Gateway** (Go) – принимает файлы от фронтенда, публикует задачи в RabbitMQ, предоставляет статус задач через Redis/PostgreSQL.
- **Worker** (Python) – асинхронно обрабатывает задачи, выполняет инференс и сохраняет результат.
- **Inference API** (FastAPI) – дополнительный синхронный доступ к модели (для тестирования и внутренних интеграций).
- **PostgreSQL** – постоянное хранение результатов оптимизации.
- **Redis** – кеширование статусов и результатов.
- **RabbitMQ** – очередь задач для асинхронной обработки.
- **Prometheus + Grafana** – мониторинг и визуализация метрик.

**Соответствие архитектурной диаграмме (ЛР1-ЛР2):**  
Реализованные компоненты полностью соответствуют спроектированной архитектуре.

### Шаг 3. Внедрение базового мониторинга

Реализованы все требуемые механизмы:

| Требование | Реализация |
|-----------|------------|
| **Логирование запросов** | Gateway логирует каждый входящий запрос (метод, путь, длительность). Worker логирует этапы обработки задач. |
| **Метрики доступности** | `up{job="gateway"}`, `up{job="worker"}` собираются Prometheus. В `docker-compose.yml` настроены healthcheck для всех сервисов. |
| **Сбор метрик нагрузки** | Для Gateway и Worker экспортируются: <br> - `http_requests_total` / `worker_tasks_processed_total` <br> - `http_request_duration_seconds` / `worker_task_duration_seconds` <br> - `worker_tasks_in_progress` |
| **Экспорт метрик** | Prometheus скрейпит `/metrics` у Gateway (порт 8080) и Worker (порт 8001). Grafana визуализирует дашборд с ключевыми показателями. |

Дашборд Grafana отображает:
- Историю обработки задач (успешно/с ошибкой)
- Длительность обработки (p95)
- Количество активных задач

### Шаг 4. Контейнеризация и оркестрация

Каждый компонент упакован в собственный Docker-образ с помощью `Dockerfile`.  
В корне проекта находится `docker-compose.yml`, описывающий все сервисы, их зависимости, переменные окружения, сеть и тома.

**Проверка запуска:**  
```bash
docker-compose up --build
```
После выполнения команды все сервисы стартуют и становятся доступны.


**Состав `docker-compose.yml`:**
- `postgres`, `redis`, `rabbitmq` – инфраструктурные сервисы
- `inference` – синхронный API (порт 8000)
- `gateway` – приём задач (порт 8080)
- `worker` – асинхронная обработка (2 реплики)
- `frontend` – веб-интерфейс (порт 80)
- `prometheus` – сбор метрик (порт 9090)
- `grafana` – визуализация (порт 3000)

**Минимальные требования (из README.md):**
- Docker 20.10+
- Docker Compose 2.0+
- Свободные порты: 80, 8080, 8000, 9090, 3000, 5432, 6379, 5672, 15672

- Prometheus: http://localhost:9090
